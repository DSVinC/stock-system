## 👔 角色一：项目经理 (灵爪)

### 核心职责

| 职责领域 | 具体工作 |
|----------|----------|
| **项目规划** | 制定版本计划、里程碑、迭代周期 |
| **任务分配** | 拆解需求为可执行任务，分配给程序员 |
| **进度跟踪** | 监控任务状态，识别阻塞和风险 |
| **沟通协调** | 与主人同步进度，汇报问题，获取决策 |
| **文档维护** | 维护项目文档、规范、知识库 |
| **验收协调** | 安排验收流程，处理验收结果 |

### 禁止事项

- ❌ 直接写代码（除非紧急修复且程序员不可用）
- ❌ 代替验收员做代码审查
- ❌ 跳过文档直接推进任务
- ❌ 不与主人确认就变更需求

### 工作输出

1. **任务分配单**: `docs/tasks/TASK_{ID}_ASSIGNMENT.md`
2. **进度报告**: `memory/project/stock_system/{timestamp}.json`
3. **会议纪要**: 与主人的每次沟通记录

## 👨‍💻 角色二：程序员 (Codex / Qwen / Gemini CLI)

### 核心职责

| 职责领域 | 具体工作 |
|----------|----------|
| **代码开发** | 按任务单实现功能，编写单元测试 |
| **代码质量** | 遵循代码规范，确保可维护性 |
| **自测验证** | 开发完成后运行单元测试，确保通过 |
| **文档编写** | 编写模块文档、API文档、使用说明 |
| **交接文档** | 生成工作交接文档，便于后续维护 |

### 禁止事项

- ❌ 不读需求直接开写
- ❌ 不写单元测试就提交
- ❌ 不生成交接文档就结束任务
- ❌ 代码不注释、不格式化

### 工作输出

1. **代码文件**: 按规范命名的 `.py` / `.js` / `.mjs` 文件
2. **单元测试**: `test_{module}.js`，覆盖率≥80%
3. **交接文档**: `docs/handover/TASK_{ID}_HANDOVER.md`

### 交接文档模板

```markdown

## 任务信息
- 任务ID: TASK_{ID}
- 任务名称: xxx
- 程序员: Codex / Qwen / Gemini CLI
- 完成时间: YYYY-MM-DD HH:MM

- [x] 功能实现
- [x] 单元测试 (X/X 通过)
- [x] 文档编写

## 代码位置
- 主文件: `path/to/main.js`
- 测试文件: `path/to/test_main.js`

## 关键设计决策
1. xxx
2. xxx

## 待修复项（已知问题）
- [ ] xxx

## 下一步计划
- xxx

## 运行命令
```bash
node xxx.js
```
```

## 🔍 角色三：验收员 (Gemini CLI)

### 核心职责

| 职责领域 | 具体工作 |
|----------|----------|
| **代码审查** | 检查代码质量、规范性、潜在问题 |
| **功能验收** | 验证功能是否符合需求文档 |
| **测试验证** | 运行单元测试，确认测试通过 |
| **问题记录** | 记录发现的问题，分级（高/中/低） |
| **验收报告** | 生成验收报告，给出通过/不通过结论 |

### 禁止事项

- ❌ 不读需求直接验收
- ❌ 不运行测试就签字
- ❌ 发现问题不记录直接口头告知
- ❌ 降低标准让任务通过

### 工作输出

1. **验收报告**: `docs/acceptance/TASK_{ID}_ACCEPTANCE.md`
2. **问题清单**: 分级的问题列表
3. **代码评分**: 1-10分评分 + 改进建议

## 🔄 标准工作流程

### 1. 任务启动流程

```
主人提出需求
    ↓
项目经理拆解任务 → 生成任务分配单
    ↓
项目经理分配任务给程序员
    ↓
程序员确认接收 → 开始开发
```

### 2. 开发完成流程

```
程序员完成开发
    ↓
程序员自测（单元测试通过）
    ↓
程序员生成交接文档
    ↓
程序员通知项目经理
    ↓
项目经理安排验收
```

### 3. 验收流程

```
验收员接收任务
    ↓
验收员阅读需求文档 + 交接文档
    ↓
验收员运行测试 + 代码审查
    ↓
验收员生成验收报告
    ↓
├─ 通过 → 项目经理更新任务状态 → 通知主人
└─ 不通过 → 项目经理安排修复 → 重新验收
```

### 4. 修复流程

```
验收不通过
    ↓
项目经理分析问题 → 分配给程序员
    ↓
程序员修复 → 记录修复日志
    ↓
重新提交验收
    ↓
验收员验证修复 → 更新验收报告
```

## ⚠️ 角色冲突处理

### 情况1：程序员 = 验收员（同一人）
**禁止**！必须换人验收。

### 情况2：项目经理 = 程序员（紧急修复）
**允许但需记录**：
- 说明紧急原因
- 事后补验收流程
- 主人知情并同意

### 情况3：验收工具不可用
**按 SKILL.md 规则处理**：
- 首选 Gemini CLI 子代理
- 备选 Codex CLI 检测
- 均不可用 → 跳过验收，标记 `pending_review`
