## 👔 角色一：任务编排与验收协调 (Codex)

### 核心职责

| 职责领域 | 具体工作 |
|----------|----------|
| **需求澄清** | 先读需求、代码和上下文，再确认边界、约束与验收标准 |
| **任务拆分** | 将需求拆成可并行执行的子任务，明确依赖关系与交付物 |
| **并行编排** | 为 subagent 分配任务，必要时把局部实现交给 Claude Code 作为可选开发器 |
| **进度跟踪** | 监控任务状态、阻塞点、风险和回归影响 |
| **结果验收** | 基于代码变更、测试结果和交接文档完成最终验收判断 |
| **状态同步** | 负责与主人同步进度、问题、结论和 todo.db 状态更新 |
| **文档维护** | 维护任务单、交接记录、验收结论和流程文档 |

### 禁止事项

- ❌ 不读需求和现有上下文就直接拆任务或下发实现
- ❌ 把本可以并行的工作串行化，浪费 subagent 并行能力
- ❌ 让 Gemini 充当默认主开发或默认最终验收者
- ❌ 把 Claude Code 当成默认主力，而不是按需使用的可选实现器
- ❌ 跳过交接、测试或验收记录直接宣布完成
- ❌ 未确认就擅自变更需求边界或验收口径

### 工作输出

1. **任务拆解单**: `docs/tasks/TASK_{ID}_ASSIGNMENT.md`
2. **并行执行计划**: 子任务列表、依赖关系、责任分配
3. **验收结论**: 通过 / 有条件通过 / 不通过 + 处理意见
4. **同步记录**: `memory/project/stock_system/{timestamp}.json`
5. **状态更新**: todo.db / 任务状态流转记录

## 👨‍💻 角色二：并行实现者 (subagent / Claude Code 可选)

### 核心职责

| 职责领域 | 具体工作 |
|----------|----------|
| **代码开发** | 按 Codex 拆分的子任务实现功能，优先并行完成独立模块 |
| **代码质量** | 遵循代码规范，确保可维护性 |
| **自测验证** | 开发完成后运行相关测试，确保变更可回归 |
| **文档编写** | 编写模块文档、API文档、使用说明 |
| **交接文档** | 生成工作交接文档，便于后续维护 |
| **局部修复** | 对 Codex 返修单进行定点修复，保持变更最小化 |

### 禁止事项

- ❌ 不读需求直接开写
- ❌ 不写单元测试就提交
- ❌ 不生成交接文档就结束任务
- ❌ 不按任务单擅自扩大修改范围
- ❌ 把 Claude Code 当成默认主开发器，脱离 Codex 编排单独推进
- ❌ 代码不注释、不格式化、不做最小必要验证

### 工作输出

1. **代码文件**: 按规范命名的 `.py` / `.js` / `.mjs` 文件
2. **单元测试**: `test_{module}.js`，覆盖率≥80%
3. **交接文档**: `docs/handover/TASK_{ID}_HANDOVER.md`
4. **修复说明**: 若有返修，需附带变更点和验证结果

### 调用方式

**✅ 正确方式：由 Codex 按任务编排后并行调用**
```bash
cd /Users/vvc/.openclaw/workspace/stock-system && \
<subagent-or-claude-code-command> &
```

**⚠️ 配置要求**:
- 以 Codex 的任务拆解结果为准，接收明确的子任务边界
- 需要时由 Codex 提供环境变量、测试目标和验收口径
- 长任务可后台执行，但必须保留可追踪输出

**❌ 禁止方式**:
- 不要绕过 Codex 直接接收主人口头需求就开始实现
- 不要把一次子任务实现扩展成跨模块重构
- 不要默认抢占所有实现工作，Claude Code 仅在需要时作为补充

## 🔍 角色三：检索 / PR / review 辅助 (Gemini CLI)

### 核心职责

| 职责领域 | 具体工作 |
|----------|----------|
| **search 辅助** | 快速查找仓库、历史实现、相似问题和候选修复方向 |
| **PR 辅助** | 协助整理 PR 描述、变更摘要、检查清单和发布说明 |
| **review 辅助** | 协助做代码 review、指出风险、遗漏和边界情况 |
| **验收辅助** | 在 Codex 组织的验收流程中提供补充检查，而不是默认主验收者 |
| **问题归档** | 整理发现的问题、建议和跟踪项，便于复验闭环 |
| **检索结论** | 输出可引用的搜索结果、对比结论和待确认事项 |

### 禁止事项

- ❌ 不把自己设为默认主开发或默认主验收
- ❌ 不运行必要检查就给出“通过”结论
- ❌ 发现问题不记录直接口头告知
- ❌ 为了推进进度而降低 review 标准
- ❌ 在没有 Codex 归档的情况下单独完成状态收口

### 工作输出

1. **检索摘要**: 候选实现、关联文件、参考结论
2. **PR/review 备注**: 风险点、建议修改、待确认项
3. **辅助验收记录**: 供 Codex 汇总成最终验收结论
4. **问题清单**: 分级的问题列表

## 🔄 标准工作流程

### 1. 任务启动流程

```
主人提出需求
    ↓
Codex 读取需求、上下文和历史记录
    ↓
Codex 拆解任务 → 生成任务分配单和并行执行计划
    ↓
Codex 分配子任务给 subagent / 必要时分配给 Claude Code
    ↓
执行前确认验收口径和同步要求
```

### 2. 开发完成流程

```
subagent 并行完成开发
    ↓
各实现者自测并补交交接文档
    ↓
Codex 汇总变更、检查测试和边界影响
    ↓
Codex 组织最终验收
    ↓
必要时调用 Gemini 做 search / PR / review 辅助
```

### 3. 验收流程

```
Codex 接收验收请求
    ↓
Codex 阅读需求文档 + 交接文档 + 代码变更
    ↓
Codex 完成测试验证 + 代码审查 + 状态核对
    ↓
Codex 生成验收结论
    ↓
├─ 通过 → Codex 同步 todo.db / 状态 → 通知主人
└─ 不通过 → Codex 拆返修项 → 分配 subagent / Claude Code 修复
```

### 4. 修复流程

```
验收不通过 / 发现回归
    ↓
Codex 整理问题单与修复边界
    ↓
subagent / Claude Code 定点修复 → 记录修复说明
    ↓
重新提交给 Codex
    ↓
Codex 复核修复结果并决定是否进入复验
```

### 5. 重新验收流程（闭环规范）✅

```
验收不通过/有条件通过
    ↓
实现者修复问题
    ↓
生成修复报告 (docs/fixes/TASK_{ID}_FIX_{N}.md)
    ↓
更新任务状态为 `pending_review`
    ↓
通知 Codex 安排复验
    ↓
Codex 执行复验（只验证修复项 + 回归测试）
    ↓
├─ 全部通过 → 更新验收报告为「通过」→ 任务关闭
└─ 仍有问题 → 更新验收报告 → 继续修复循环
```

**复验规范**:
1. **修复报告必须包含**:
   - 修复的问题清单（对应验收报告中的问题 ID）
   - 修复方案说明
   - 修改的文件列表
   - 回归测试结果

2. **Codex 复验重点**:
   - 验证修复项是否真正解决
   - 运行回归测试确保无新 bug
   - 检查修复是否引入新问题

3. **循环终止条件**:
   - Codex 验收结论为「通过」
   - 或主人特批「有条件通过」

4. **状态流转**:
   ```
   in_progress → pending_review → in_review → done
                          ↑           ↓
                          └──────←────┘
                         (循环直到通过)
   ```

**修复报告模板**:
```markdown
# 修复报告

## 任务信息
- 任务 ID: TASK_{ID}
- 修复版本：FIX_{N}
- 修复时间：YYYY-MM-DD HH:MM
- 修复人：xxx

## 修复的问题

| 问题 ID | 问题描述 | 修复方案 | 修改文件 |
|---------|----------|----------|----------|
| #1 | xxx | xxx | `path/to/file.js` |
| #2 | xxx | xxx | `path/to/file.js` |

## 回归测试
- [ ] 单元测试通过 (X/X)
- [ ] 集成测试通过 (X/X)
- [ ] 手动验证通过

## 已知遗留问题
- [ ] xxx（如有，需说明原因和后续计划）

## 申请复验
- [x] 修复完成，申请复验
```

## ⚠️ 角色冲突处理

### 情况1：实现者 = 复验者（同一人）
**尽量避免**；如确需同一实现者修复，最终复验必须由 Codex 完成。

### 情况2：Codex = 实现者（紧急修复）
**允许但需记录**：
- 说明紧急原因
- 事后补复验流程
- 主人知情并同意

### 情况3：Gemini 不可用
**按 SKILL.md 规则处理**：
- 改用 Codex 直接完成 search / review / PR 辅助
- 不影响最终验收由 Codex 负责
- 仅保留记录，不切换主流程
