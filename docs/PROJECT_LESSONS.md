## 2026-03-28 | 结构化结果字段不要直接 `Number(value)`，先过滤空值避免把 null 误写成 0

### 问题类型
**边界值处理 / 结果快照准确性** - 对可选数值字段直接 `Number(value)` 会把 `null` 转成 `0`，导致状态摘要出现“看似有效、实际错误”的数值

### 背景
V5_007 的 Optuna trial 结果在 `resultSummary` 沉淀时，`completedTrials` 原本是可选字段。代码若直接走 `Number(value)`，`null` 会被转换成 `0`，前端会错误展示“完成 0 次试验”。

### 关键经验
1. 结果快照里的可选数值字段必须先做空值判定
2. `null/undefined/''` 应保持“无值”语义，而不是变成 `0`
3. 这类问题要靠接口级断言和 UI 展示断言双重覆盖

### 预防措施
1. 统一使用 `parseOptionalFiniteNumber` 这类 helper 做可选数值归一化
2. 对任务摘要字段新增测试断言（例如 `requestedTrials/completedTrials/trialCount`）
3. 回归时同时检查“后端快照值”和“前端展示值”是否一致

## 2026-03-28 | 跨语言优化器先打通“真实评分 CLI”，不要在 Python 里重写回测

### 问题类型
**架构收口 / 跨语言集成** - 当 Python 优化器需要依赖现有 Node 回测能力时，最小闭环应先做“可调用的真实评分 CLI”，而不是把回测逻辑搬两份

### 背景
V5_007 的 `scripts/optuna_optimizer.py` 原本只有：
- `# TODO: 调用回测 API 计算评分`
- `score = 75.0`

如果直接在 Python 里重写回测和评分，不仅工作量大，还会把 Node 和 Python 的策略语义逐步拉开。

### 关键经验
1. **最短真实闭环 = Node CLI 负责回测评分，Python 负责搜索算法**
2. **跨语言边界要只传必要输入/输出**：
   - 输入：策略类型、股票、日期区间、参数 JSON
   - 输出：`success / scoreTotal / level / metrics / params`
3. **第一版不要把版本保存、任务持久化、优化历史都塞进去**，否则最小切口会迅速膨胀

### 预防措施
1. 以后再做 Python/Node 混合能力时，优先先做一个最小 CLI 契约
2. 真实评分链路打通前，不要再接受“常量占位分数”留在主干
3. 对双均线这类策略，跨语言参数名要明确映射，避免 `ma_short/ma_long` 与 `fast_period/slow_period` 漂移

## 2026-03-28 | 环境依赖补齐后，要立刻补一条最小真实 smoke，别停在“代码看起来对”

### 问题类型
**验收策略 / 环境收口** - 代码层把占位逻辑移除后，如果环境依赖已经能补齐，就应该马上补一条最小真实执行，而不是停在结构性测试

### 背景
V5_007 在代码层已经从 `75.0` 假分数改成真实评分 CLI，但如果不继续装 `optuna` 并跑一条最小 trial，状态仍然会卡在“理论上可运行”。

### 关键经验
1. **依赖补齐后，最值钱的是一条最小真实 smoke**
2. **最小真实 smoke 不求覆盖面，只求确认主链能跑通**
3. **这样能把状态从“代码闭环”推进到“环境可运行闭环”**

### 预防措施
1. 对“代码已闭环但环境缺依赖”的任务，优先补齐依赖并跑 1 条最小样本
2. 报告里区分清楚：
   - 最小可运行闭环
   - 大规模/正式生产闭环
3. 不要把 smoke 的单次样本结果误写成策略效果结论

## 2026-03-28 | 新后端能力先走“后端接入 -> 页面开关”两步，不要一次把研究流 UI 和执行链路一起改

### 问题类型
**任务拆分 / 研究流接入** - 真实优化能力从脚本接到页面时，最稳的推进方式是先让后端路由可用，再补前端选择器

### 背景
V5_007 真实优化已经具备最小 CLI 与 Optuna smoke。如果下一步同时改：
- `iteration-manager` 后端分支
- 页面请求体
- 页面展示/提示

就会把问题来源混在一起，不利于验收和回滚。

### 关键经验
1. **先接后端，再开页面入口，验收最清楚**
2. **默认路径保持不变很重要**：`heuristic` 仍应是默认值，避免新能力影响旧链路
3. **新字段要从启动请求一路写到快照与状态回包**，否则页面恢复链路会断

### 预防措施
1. 新增“可选后端/引擎”能力时，默认值必须兼容旧路径
2. 后端接入完成后，先补接口级测试，再补页面选择器
3. 页面层只负责切换和可见提示，不在同一轮追加额外复杂逻辑

## 2026-03-28 | 页面选择器接入新后端时，先提交显式字段，再谈浏览器联动

### 问题类型
**前端接入 / 回归策略** - 当页面要暴露一个新的执行后端时，第一轮最重要的是让请求体显式、默认值稳定，而不是一开始就补复杂交互

### 背景
`iteration-manager` 需要把 `optuna` 正式暴露给研究流页面。如果第一轮就同时加联动提示、更多状态文案和浏览器级行为，很容易把问题源头混在一起。

### 关键经验
1. **先让请求体稳定带字段，页面就算接入成功了一半**
2. **默认值必须可读、可见、可回归**：`heuristic` 仍是默认快速模式
3. **页面测试先覆盖“默认值 + 显式切换”两种请求体**，浏览器级联动放下一轮

### 预防措施
1. 新增前端模式选择器时，先补 Node harness 断言请求体
2. 页面选择器文案要直接说明模式差异，避免用户误选
3. 浏览器联动回归单独成任务，不和字段接入混在同一轮

## 2026-03-28 | URL 导入的研究输入不能只显示在摘要里，表单控件也要同步

### 问题类型
**前端状态同步 / 浏览器回归** - 页面如果一边从 URL 渲染研究输入摘要，一边又从表单控件提交请求，就必须保证两处状态同步，否则会出现“看上去是 A，提交出去是 B”

### 背景
`iteration-manager` 的浏览器 smoke 首轮失败暴露出真实问题：
- research summary 正确显示了 `parallelTasks=6`
- 但 `#parallelTasks` 输入框仍保持默认 `4`
- 用户点击开始后，请求体带出去的还是 `4`

### 关键经验
1. **显示摘要不等于真正生效**
2. **凡是 URL 导入后还会从表单取值的字段，都要把表单一并同步**
3. **这种问题最容易被浏览器级 smoke 抓住，纯结构测试容易漏掉**

### 预防措施
1. URL 导入字段默认做“双写”：摘要显示 + 表单赋值
2. 对关键输入字段，至少保留一条真实页面请求体 smoke
3. 如果页面存在“摘要视图”和“编辑控件”双状态，后续修改时优先检查它们是否漂移

## 2026-03-28 | 可选后端字段要把默认值显式化，否则恢复链路会出现“信息缺失”

### 问题类型
**响应口径一致性 / 恢复可见性** - 同一字段如果在一个分支返回、另一个分支省略，前端恢复视图会出现“未导入”假象

### 背景
`iteration-manager` 在 `optuna` 路径会回显 `optimizationBackend`，但默认 `heuristic` 路径原先可能省略该字段，导致前端摘要显示不完整。

### 关键经验
1. 可选能力字段也应该有稳定默认值
2. “省略字段”在恢复视图里会被解释成“数据缺失”，而不是“默认模式”
3. 后端统一显式回传，比前端做猜测更稳

### 预防措施
1. 对 `optimizationBackend` 这类模式字段，默认值也显式返回
2. 新字段接入后补“默认路径”回归测试，避免只测 feature path
3. 恢复类页面优先消费标准化后的摘要对象

## 2026-03-28 | 恢复态同步要同时更新摘要和控件，不然 UI 会出现“口径漂移”

### 问题类型
**前端状态同步 / 恢复交互** - 恢复最近任务时，页面如果只刷新摘要而不刷新控件，用户看到的状态会和下一次提交不一致

### 背景
`iteration-manager` 恢复任务时已经能显示 `optimizationBackend` 摘要，但下拉框可能还停留在默认值。这样用户会看到“摘要是 optuna、控件还是 heuristic”，下一次点击就会造成误解。

### 关键经验
1. 恢复态要双写：摘要显示 + 控件同步
2. 摘要和控件都是状态源，不能只修一边
3. 这类问题最好直接补恢复态测试，不要只靠人工看页面

### 预防措施
1. 页面恢复任务时，凡是会影响下一次提交的字段都要同步到控件
2. 恢复链路回归测试要同时看摘要和表单值
3. 新字段接入后，恢复态测试要包含“任务恢复”和“默认值回退”两种路径

## 2026-03-28 | SQL mock 不要依赖整串 includes，真实多行查询一变格式就会假失败

### 问题类型
**测试基线 / Mock 稳定性** - 用整串 `query.includes('SELECT ...')` 匹配 SQL 时，产品代码只要改成多行或空白变化，测试就会错误地走到空分支

### 背景
`TASK_TEST_004` 的性能测试里，`HistoricalDecisionEngine.preloadPrices()` 已正常工作，但 `tests/test-performance.js` 的 mock 仍然用脆弱的完整子串匹配：
- `SELECT ts_code, trade_date, close FROM stock_daily`

真实 SQL 带换行和缩进后，这个分支没有命中，导致：
- 预加载 0 条价格记录
- 缓存有效性失败
- 缓存命中率错误降到 0%

### 关键经验
1. **SQL mock 要按语义匹配，不要按排版匹配**
2. **多行 SQL 场景先做空白归一化，再判断关键片段**
3. **如果一个性能/缓存测试突然全红，先查 mock 是否失配，再怀疑产品实现**

### 预防措施
1. 测试里匹配 SQL 时，先 `replace(/\s+/g, ' ').trim()` 做归一化
2. 匹配条件优先拆成几个关键片段：
   - `FROM stock_daily`
   - `ts_code IN`
   - `trade_date BETWEEN ? AND ?`
3. 修改数据访问层 SQL 后，优先重跑默认 runner，避免 mock 静默失效太久

## 2026-03-27 | 真实 schema 漂移修复时，先写“错误依赖列”的复现测试，再迁到 side table

### 问题类型
**数据库 / 修复流程** - 导入路径依赖了主表里并不存在的列时，先把错误依赖钉住，再做 schema 对齐修复

### 背景
`api/report-storage.js#importToOrderFromReport()` 原本往 `conditional_order` 写：
- `remark`

但真实数据库主表没有这列，而且主表还要求：
- `end_date`
- `created_at`
- `updated_at`

如果直接改 SQL，很容易修掉一个问题又引入另一个时间字段问题。

### 关键经验
1. **schema 漂移修复不要只看代码，要对照真实数据库 schema**
2. **先写“错误依赖列”的复现测试，能避免修复时把旧问题说不清**
3. **报告来源 / 备注类上下文优先进入 side table，不再反向污染主表**

### 预防措施
1. 任何条件单写入路径修复前，先跑 `.schema conditional_order`
2. 对“代码写了但主表没有”的列，先建复现测试，再修
3. 修完后至少校验两件事：
   - 主表 SQL 不再引用漂移列
   - side table 正确接收上下文

## 2026-03-27 | schema 漂移场景下，优先用 side table 承接执行上下文

### 问题类型
**数据建模 / 功能开发** - 当交易主表已经存在 schema 漂移时，新增执行上下文不应继续堆到主表

### 背景
在 `TASK_FLOW_REFACTOR_006C2` 中，需要为条件单记录策略身份字段，支持后续“执行反馈 -> 策略管理”回流。检查发现：
- `conditional_order` 真实表结构中没有 `report_id / remark`
- 但代码里已有路径在尝试写这些字段
- 说明主表已经存在 schema 漂移

### 关键经验
1. **主表已有漂移时，先不要继续扩主表**
2. **执行上下文更适合 side table**：
   - 主交易记录保持稳定
   - 策略身份 / 版本 / 报告来源等上下文独立演进
3. **side table 更适合后续研究回流**
   - 后面要追加 `strategy_id / version / report_id / feedback metrics` 时，不会反复改动主表

### 预防措施
1. 新增执行/分析上下文字段前，先审计主表真实 schema 与代码写入是否一致
2. 一旦主表出现漂移，优先考虑 side table，而不是继续往主表塞字段
3. side table 第一轮只做最小写入路径，读取聚合与其他写入路径拆成后续子任务

## 2026-03-27 | HTML 异步函数测试不能只过结构断言，必须防止“假通过”

### 问题类型
**代理协作 / 测试策略** - HTML 内嵌 `async function` 的回归测试，容易出现“代码结构命中但运行时没有真正执行”的假通过

### 背景
在 `TASK_FLOW_REFACTOR_006C1` 中，需要验证 `conditional-order.html` 的 `executeBatchCreate()` 是否把策略身份字段透传到批量条件单的请求体。首轮候选虽然补了结构断言，但运行时 harness 没有真正等待异步 IIFE 执行，导致：
- 结构检查通过
- 运行时 `fetchBodies` 仍然是空数组

### 关键经验
1. **结构断言只能证明“代码里写了字段”，不能证明逻辑真的执行到了 fetch**
2. **HTML 内嵌异步函数测试要特别小心执行包装层**：如果外层没有 `return/await`，就会得到假通过或半通过
3. **验收时必须同时看两层**：
   - 结构层：目标字段是否真的写进对象
   - 运行层：关键副作用是否真的发生

### 预防措施
1. HTML 异步函数测试默认要求“双层验收”：结构断言 + 运行时断言
2. 如果运行时 harness 不稳定，至少要在状态文档里明确“当前只覆盖结构级回归”
3. 代理首轮只交结构级测试时，不能直接判通过，必须继续追到运行时或明确降级为结构回归

### 问题类型
**代理协作 / 模型选型** - Claude Code 后端模型不能只看探针通过，要看真实小任务命中率

### 背景
为了给 Claude Code 选择更合适的临时模型，对同一个小任务做了三模型 A/B：
- `glm-5`
- `qwen3-coder-plus`
- `MiniMax-M2.5`

任务内容是研究流页面文案收口，仅涉及 `backtest.html` 与 `iteration-manager.html` 的 5 处精确文案替换。

### 结果对比

| 模型 | 命中率 | 观察 |
|------|--------|------|
| `glm-5` | 5/5 | 一轮全命中，文案与要求一致 |
| `qwen3-coder-plus` | 2/5 | 能理解方向，但主说明文案连续简化，未按要求落成目标版本 |
| `MiniMax-M2.5` | 2/5 | 与 `qwen3-coder-plus` 表现接近，倾向生成“合理改写”而不是“按要求命中” |

### 关键经验
1. **探针通过不等于适合主开发**：6 个模型都能返回 `OK`，但真实小任务命中率差异很大
2. **当前项目的小范围精确收口任务，`glm-5` 更稳**
3. **`qwen3-coder-plus` / `MiniMax-M2.5` 更像“会改写出一个看起来合理的版本”，但不一定命中指定文案**

### 预防措施
1. Claude Code 主开发默认保留 `glm-5` 作为第一候选
2. 新模型切换前，先做同任务 A/B，再决定是否替换主模型
3. A/B 任务统一使用：
   - 独立候选工作区
   - 同一 prompt
   - 同一评分标准
4. 真实落主项目前，先从候选结果里选冠军，再做一次逐字纠偏验收

## 2026-03-27 | HTML 页面逻辑 bug 先做片段级复现测试，再做多模型修复

### 问题类型
**代理协作 / 测试策略** - 前端页面脚本埋在 HTML 中时，仍然可以先做函数级复现测试，不必跳过“先复现再修”

### 背景

// 1262 次 = 25 分钟

## 2026-03-26: V4 选股功能权重配置交付问题

### 问题描述

V4 阶段交付的四维度七因子策略参数面板，主人调整权重后选股结果无变化。权重配置 UI 正常、API 传递正常，但后端评分逻辑使用固定值，未使用传入的权重动态计算。

### 根因分析

**直接原因**：
1. `stock_factor_snapshot` 表只有预计算的 `seven_factor_score`，没有单独的七个因子原始数据列
2. 后端评分代码使用固定权重：`dimensions: { social: 3, policy: 3, public: 3, business: 3 }`
3. 验收测试只验证了 UI 和流程，没有验证"调整权重→结果变化"的业务逻辑

**深层原因**：
1. **设计阶段**：没有验证数据结构是否支持设计功能，凭记忆假设数据存在
2. **开发阶段**：满足于表面完成（UI 有了、API 通了），没有实现核心逻辑
3. **验收阶段**：只测试了"有结果"，没有测试"结果随参数变化"
4. **沟通不足**：没有主动说明"权重配置暂不影响选股结果"的限制

### 修复方案

1. **数据库迁移**：添加 11 个新列（7 个因子 + 4 个维度）
   ```bash
   node scripts/migrate_add_factor_columns.js
   ```
2. **数据回填**：从现有数据计算七个因子原始值
   ```bash
   node scripts/backfill_seven_factors.js --auto
   ```
3. **API 修改**：使用动态权重计算评分
   ```javascript
   // 使用传入的权重动态计算
   const w = filters.factorWeights;
   dynamicScore = (
     row.trend_score * w.trend +
     row.momentum_score * w.momentum +
     // ...
   );
   ```

### 影响范围

- **受影响功能**：V4 选股功能（四维度七因子策略）
- **影响时间**：2026-03-25 交付至 2026-03-26 发现（约 9 小时）
- **影响程度**：权重配置功能不可用，但筛选条件（PE、最低评分）可用

### 验收测试补充

**新增测试用例**：
```
1. 默认权重 → 选股结果 A
2. 调整权重（如 trend=0.30, valuation=0.05） → 选股结果 B
3. 验证 A ≠ B
4. 验证 B 中趋势因子得分高的股票排名上升
```

### 经验总结

**核心教训**：不是技术能力问题，是工程纪律问题

**三个错误做法**：
1. ❌ 假设数据存在，没有查询确认
2. ❌ 满足于表面完成，没有端到端验证
3. ❌ 害怕暴露限制，没有主动说明

**三个正确做法**：
1. ✅ 先验证，再承诺，后交付
2. ✅ 功能经测试验证正常，不只是"看起来正常"
3. ✅ 明确告知"什么能用"和"什么暂不能用"

**口诀**：
> 数据驱动，不是假设驱动！
> 端到端测试，不是表面测试！
> 透明沟通，不是隐瞒限制！

**复用场景**：
- 所有涉及数据结构扩展的功能开发
- 所有需要动态配置影响业务逻辑的功能
- 所有有"看起来能用但实际不能用"风险的功能

## 2026-03-27: 执行反馈表优先沉淀“事件”，不要污染交易主表

### 背景

- 能不能用侧表/事件表承接

- 能不能保持主交易表 schema 稳定

- 避免因为评分权重没定好而反复改 schema / API

- `position_closed_count`
- `win_rate`

- `avg_realized_return`
- `avg_holding_days`
- `trigger_failure_count`
- `trigger_failure_rate`

## 2026-03-27: 样本少的时候，先给“状态/置信度”，不要急着改总分

### 背景

执行反馈刚接进研究流时，最大的风险不是没有指标，而是“样本太少却被当成强结论”。如果这个时候直接改 `backtest_score`，很容易把排序搞偏。

### 当前采用的做法

第一版先增加两个辅助字段：

- `execution_feedback_status`
- `execution_feedback_confidence`

让研究流页面先能表达：

- 倾向是正向、混合、还是需要警惕
- 这个判断的样本到底靠不靠谱

### 这条规则适用的场景

- 新指标刚接入评分体系
- 样本数明显不足
- 产品需要先“提示判断”，而不是立刻“自动排序”

### 页面层的落地方式

第一版优先用：

- 状态标签
- 置信度标签

而不是一开始就上复杂图表。这样更容易让用户先理解“这个版本目前看起来怎么样”与“这个判断有多可靠”。

### 第二步增强

当标签已经落地后，优先加：

- 前端筛选

而不是立刻加更重的 compare 维度扩展。这样改动面更小，用户收益更直接。

### 公共逻辑复用

当 versions 和 compare 都开始消费 feedback 时，不要复制粘贴 enrichment 逻辑。先抽公共 helper，再分别挂到两个接口，后续维护成本会明显更低。

## 2026-03-27: compare 页面第二步只展示“反馈摘要”，不要一上来改图表评分逻辑

### 背景

compare API 在带上 execution feedback 后，最容易走偏的方向是直接把这些字段塞进雷达图或改 `best` 逻辑。这样会把“展示增强”和“评分体系变更”混成一次改动，验收成本也会明显上升。

第二步只让 compare 页面新增一个独立的反馈摘要面板，显示：

- 用户能直接读到“回测版本的真实执行反馈”，但不会误以为系统已经自动重算排名
- 后续如果要继续增强，可以在已有 compare 面板上逐步扩展，而不是推翻现有图表逻辑

## 2026-03-27: 新链路的语义不能自创，必须沿用前面已经落地的状态体系

### 背景

- `success / partial / failed / pending`
- 数值型 `confidence`

这和研究流前面已经落地的：

- `no_data / positive / mixed / caution`

直接冲突。

测试先改成统一语义，再倒逼实现修正。最后发布桥和 side table 一律沿用研究流已经使用的状态体系。

### 经验

- 新功能接入现有闭环时，先查“系统里已经存在的语义规范”，不要让代理临时发明一套新枚举
- 如果代理已经写出错语义，优先改测试期望，把偏差固定为真实失败，再修实现

## 2026-03-27: 页面层消费新数据时，先做模板级验证，再补浏览器级回归

- 对于 HTML 单文件页面，模板级测试是很好的中间层

## 2026-03-28: 前端脚本抽取式测试环境下，新增 helper 要考虑缺失兜底

### 背景

部分 `iteration-manager` 测试会按函数粒度抽取脚本，不一定包含全部 helper。  
新增 `formatResultReason()` 后，`updateBestConfig()` 在这类环境里可能触发 `ReferenceError`。

### 当前采用的做法

- 在 `updateBestConfig()` 中加入 `formatResultReason` 缺失兜底
- 同步调整快照测试期望，跟随参数归一化后的新字段与新默认值

### 经验

- 对“函数抽取式”测试友好的做法是：核心渲染函数尽量自包含或有兜底
- 行为口径变更后，测试断言要同步迁移，避免“实现正确但断言过时”

## 2026-03-28: 总测偶发红灯优先做“确定性收口”，再看业务逻辑

### 背景

`run-all-tests` 中个别用例单跑通过、总跑偶发失败，通常是测试数据随机性或执行环境差异引起，而非真实业务回归。

### 当前采用的做法

- 给关键断言场景注入确定性输入（固定快照分数）
- 对函数抽取式测试补 helper 缺失兜底
- 复跑总入口确认全绿

### 经验

- “单跑过、总跑挂”先看可重复性和输入确定性
- 修总测时优先收口测试基线，不要先动业务核心逻辑

## 2026-03-28: README 只保留高层状态，详细进度必须指向单一真源

### 背景

README 中长期保留“某版本全部完成”的静态结论，很容易和运行中的 `PROJECT_PROGRESS/runtime` 口径冲突。

### 当前采用的做法

- README 改为只写高层状态与入口链接
- 明确 `docs/PROJECT_PROGRESS.md` 为进度真源
- 任务级状态一律落 `docs/runtime/{TASK_ID}_STATUS.md`

### 经验

- 进度文档应“单点写入、多处引用”，避免多份状态副本漂移
- README 适合作为导航，不适合作为实时任务看板
