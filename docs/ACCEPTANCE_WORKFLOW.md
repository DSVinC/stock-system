# 验收流程规范

> **版本**: v1.0  
> **创建**: 2026-03-20  
> **适用范围**: 所有任务验收场景

---

## 🎯 核心原则

1. **验收是质量闸门** - 不通过的任务不得进入生产
2. **问题必须闭环** - 所有问题必须有修复和复验记录
3. **循环直到通过** - 修复后必须重新验收，形成闭环
4. **文档可追溯** - 验收报告、修复报告、状态流转必须完整记录

---

```
  in_progress

       ▼

       │ 验证通过      │
       ▼              │
    done ◄────────────┘
   (闭环)
```

## 🔄 完整验收流程

### 阶段 1: 验收准备

**输入**:
- 任务分配单：`docs/tasks/TASK_{ID}_ASSIGNMENT.md`
- 交接文档：`docs/handover/TASK_{ID}_HANDOVER.md`
- 代码 PR / 修改文件

**验收员动作**:
1. 阅读需求文档和交接文档
2. 确认验收标准
3. 准备测试环境

---

### 阶段 2: 执行验收

**验收员动作**:
1. **代码审查** - 检查代码质量、规范性
2. **功能验证** - 运行功能测试
3. **单元测试** - 运行 `npm test` 或等效命令
4. **回归测试** - 确保无新 bug

**输出**:
- 验收报告：`docs/acceptance/TASK_{ID}_ACCEPTANCE.md`
- 问题清单（如有）

**验收结论选项**:
| 结论 | 适用场景 | 后续动作 |
|------|----------|----------|
| ✅ 通过 | 所有测试通过，无问题 | 任务关闭，合并代码 |
| ⚠️ 有条件通过 | 有小问题但不影响核心功能 | 记录问题，限期修复 |
| ❌ 不通过 | 有严重问题或核心功能不达标 | 返回修复，重新验收 |

---

### 阶段 3: 修复循环（如需要）

**触发条件**: 验收结论为「不通过」或「有条件通过」

#### 3.1 程序员修复

**动作**:
1. 阅读验收报告，理解问题
2. 分析原因，制定修复方案
3. 实施修复
4. 自测验证
5. 生成修复报告

**修复报告**: `docs/fixes/TASK_{ID}_FIX_{N}.md`

**修复报告内容**:
```markdown
# 修复报告

## 任务信息
- 任务 ID: TASK_{ID}
- 修复版本：FIX_{N} (N=1,2,3...)
- 修复时间：YYYY-MM-DD HH:MM
- 修复人：xxx

## 修复的问题

| 问题 ID | 问题描述 | 修复方案 | 修改文件 |
|---------|----------|----------|----------|
| #1 | xxx | xxx | `path/to/file.js` |
| #2 | xxx | xxx | `path/to/file.js` |

## 回归测试
- [ ] 单元测试通过 (X/X)
- [ ] 集成测试通过 (X/X)
- [ ] 手动验证通过

## 已知遗留问题
- [ ] xxx（如有，需说明原因和后续计划）

## 申请复验
- [x] 修复完成，申请复验
```

---

#### 3.2 更新任务状态

**动作**:
1. 更新 `docs/runtime/TASK_{ID}_STATUS.md`
2. 状态改为 `pending_review`
3. 通知项目经理安排复验

**状态更新模板**:
```markdown
## 状态更新 - {timestamp}

- **状态**: `pending_review`
- **修复版本**: FIX_{N}
- **修复摘要**: 修复了验收报告中的 X 个问题
- **修复报告**: `docs/fixes/TASK_{ID}_FIX_{N}.md`
- **申请**: 请求复验
```

---

#### 3.3 验收员复验

**动作**:
1. 阅读修复报告
2. **重点验证修复项**（不是全量回归）
3. 运行针对性测试
4. 确认问题已解决

**复验通过标准**:
- 修复的问题已解决
- 无新问题引入
- 回归测试通过

**输出**:
- 更新验收报告，添加复验记录
- 更新结论为「通过」或继续「不通过」

**复验记录模板**:
```markdown
## 复验记录 #1

- **复验时间**: YYYY-MM-DD HH:MM
- **复验员**: xxx
- **修复版本**: FIX_{N}
- **验证项**:
  - [x] 问题 #1 已修复
  - [x] 问题 #2 已修复
  - [x] 回归测试通过
- **结论**: 通过 / 不通过（仍有问题见下文）

### 遗留问题（如不通过）
| 问题 ID | 描述 | 状态 |
|---------|------|------|
| #3 | xxx | 未修复 |
```

---

### 阶段 4: 验收通过

**动作**:
1. 更新验收报告结论为「通过」
2. 更新任务状态为 `done`
3. 合并代码（如适用）
4. 通知项目经理和主人

**最终输出**:
- 验收报告（结论：通过）
- 所有修复报告（如有）
- 任务状态：`done`

## ⚠️ 常见问题处理

### Q1: 修复后仍有问题怎么办？
**A**: 继续修复循环，生成 FIX_{N+1}，直到验收通过。

### Q2: 修复引入了新问题怎么办？
**A**: 新问题加入问题清单，一并修复后重新验收。

### Q3: 多次修复仍无法通过怎么办？
**A**: 
1. 项目经理介入分析原因
2. 考虑调整方案或重新设计
3. 必要时向主人汇报，请求决策

### Q4: 有条件通过的任务如何处理？
**A**: 
1. 记录遗留问题和修复期限
2. 创建跟踪任务（如 TASK_{ID}_FIX）
3. 限期修复并复验

## 📈 验收指标

| 指标 | 计算方式 | 目标 |
|------|----------|------|
| 一次通过率 | 一次通过数 / 总验收数 | ≥ 70% |
| 平均修复轮次 | 总修复次数 / 不通过任务数 | ≤ 2 |
| 验收周期 | 验收开始到通过的时间 | ≤ 24 小时 |
| 问题闭环率 | 已修复问题数 / 总问题数 | 100% |
