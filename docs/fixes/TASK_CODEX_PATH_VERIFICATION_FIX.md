## 根因分析

| 层级 | 原因 | 违反原则 |
|------|------|---------|
| 表面 | 没有先检查文件就汇报路径 | AGENTS.md："不确定时先读文档，再行动" |
| 深层 | 把"推测"当"事实"汇报 | MEMORY.md："不确定就说'不知道'，别瞎编" |
| 根本 | 跳过"先验证再输出"的基本流程 | 项目操作规范："先确认再执行" |

---

## 改进纪律

### 纪律 1：文件路径验证纪律

**触发条件**：需要给出任何文件路径、命令、配置时

**执行流程**：
1. 先执行 `find`/`ls`/`read` 验证实际存在
2. 复制实际路径（不要手动输入）
3. 再汇报给用户

**检查清单**：
- [ ] 路径是否实际存在？
- [ ] 文件名是否完全匹配（包括大小写、连字符）？
- [ ] 目录层级是否正确？

### 纪律 2：不确定信号纪律

**触发条件**：对任何信息不确定时

**执行动作**：
- 明确说"需要先检查"
- 不编造看似合理的答案
- 给出验证所需时间预估

### 纪律 3：输出前验证纪律

**触发条件**：准备输出具体信息（路径、命令、配置、数据）时

**执行动作**：
- 停顿 3 秒自检："这个信息我验证过吗？"
- 如果没有，先验证再输出
- 如果无法验证，明确标注"未验证"
