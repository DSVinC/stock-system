## 一、核心投资理念

### 双轮驱动策略

系统采用"中长期为主 + 短期量化为辅"的双轮驱动投资策略：

| 策略类型 | 定位 | 实现功能 |
|----------|------|----------|
| **中长期** | 主力策略 | 监控面板 + 条件单 + 模拟炒股 |
| **短期量化** | 辅助策略 | 回测系统 + 技术策略优化 |

### 目标

- 通过中长期价值投资把握核心收益
- 通过短期量化波段获取超额收益
- 实现更高的整体投资回报

## 二、功能模块定位

### 2.1 监控面板 + 条件单

**定位**: 中长期投资策略验证工具

**核心逻辑**:
- 基于个股分析报告的决策意见
- 结合模拟炒股验证分析报告的有效性
- 面向中长期投资周期

**决策意见来源**:
1. 基本面变化
2. 止损价位
3. 止盈价位
4. 进入建仓区间

### 2.2 回测系统

**定位**: 短线操作 + 多股票量化策略优化工具

**核心逻辑**:
- 针对短线操作策略
- 多股票组合回测
- 技术指标策略参数优化

### 2.3 模拟炒股

**定位**: 策略验证工具

**核心逻辑**:
- 验证个股分析报告意见是否科学合理
- 实盘前的策略验证沙盒

## 三、条件单重构设计

### 3.1 当前问题

当前条件单只支持纯技术面条件：
- 价格、涨跌幅、量比、RSI、MACD、PE 百分位、主力净流入

**问题**: 缺乏与个股分析报告决策意见的关联

### 3.2 重构方向

条件单应基于个股分析报告的决策意见，而非纯技术指标：

| 决策类型 | 报告字段 | 条件单动作 |
|----------|----------|------------|
| 止损 | 硬止损/动态止损价位 | 股价 <= X 时卖出/提醒 |
| 止盈 | 第一目标/第二目标 | 股价 >= X 时卖出/提醒 |
| 建仓 | 建仓位置/建仓条件 | 股价进入区间时买入/提醒 |
| 加仓 | 加仓条件 | 条件满足时加仓 |
| 观察事件 | 关键观察点 | 事件发生时提醒 |
| 基本面变化 | 财报/业绩预期 | 数据更新时提醒 |

### 3.3 条件单生成时机

**采用方案 C**: 用户在条件单页面选择"从分析报告导入"

**流程**:
```
条件单页面 → 点击"从分析报告导入" → 选择股票 → 读取报告 JSON → 提取决策意见 → 自动生成条件单
```

## 七、条件单与模拟账户联动逻辑

### 7.1 触发流程

条件单被触发时，执行以下流程：

```
条件单触发
    │
    ├─→ 飞书推送（通知主人）
    │
    └─→ 模拟账户自动执行（买入/卖出）
        │
        ├─→ 记录交易明细
        ├─→ 计算交易成本（手续费、税费）
        ├─→ 更新持仓
        └─→ 计算持仓盈亏
```

**关键原则**：推送和模拟操作**同步进行**，不需要主人确认后才执行。

### 7.2 模拟账户功能

| 功能 | 说明 |
|------|------|
| **交易明细记录** | 每次买入/卖出操作都完整记录 |
| **成本计算** | 计算 A 股相关手续费、税费、分红 |
| **持仓管理** | 实时更新持仓数量和成本价 |
| **盈亏追踪** | 根据持仓情况实时计算浮动盈亏 |
| **数据存储** | 所有数据存入数据库 |

### 7.3 成本计算规则

| 费用类型 | 计算方式 |
|----------|----------|
| **佣金** | 成交金额 × 0.025%（最低 5 元） |
| **印花税** | 卖出金额 × 0.05%（买入不收，2023.8 减半） |
| **过户费** | 成交金额 × 0.002%（沪市） |
| **分红税** | 按持股时间分档（1 月内 20%，1 年 -1 月 10%，1 年以上 0%） |

## 十、策略迭代自动化流程（2026-03-23 新增，2026-03-31 修订）⭐

### 10.1 完整自动化流程（闭环设计）

```
┌─────────────────────────────────────────────────────────────────────┐
│                        策略迭代完整闭环                              │
└─────────────────────────────────────────────────────────────────────┘

回测系统
   ↓
策略自迭代（一键自迭代）
   ↓
产生合格策略方案（评分≥0.75）
   ↓
应用到选股（个股报告产生操作建议）⭐
   │
   ├─ 基于策略参数生成选股结果
   ├─ 生成个股分析报告
   └─ 报告包含：建仓价/止损价/止盈价/操作建议
   ↓
报告导入条件单
   │
   ├─ 从分析报告提取决策意见
   ├─ 自动生成条件单
   └─ 设置触发参数（止损/止盈/建仓）
   ↓
进入监控池
   │
   ├─ 监控池跟踪持仓
   └─ 实时数据监控
   ↓
实盘数据模拟交易
   │
   ├─ 条件单触发
   ├─ 模拟账户自动执行
   └─ 记录交易明细 + 绩效追踪
   ↓
验证策略实际表现
   │
   ├─ 模拟周期：≥2 周
   ├─ 模拟收益 vs 回测收益偏差 <20%
   └─ 模拟回撤 ≤ 回测回撤
   ↓
反馈到回测系统⭐
   │
   ├─ 模拟绩效数据回传
   ├─ 与回测结果对比分析
   └─ 识别偏差原因
   ↓
二次迭代（形成闭环）⭐
   │
   ├─ 方案 A：调整策略参数
   │   ├─ 回撤过大 → 收紧风控阈值
   │   ├─ 胜率过低 → 放宽入场条件
   │   └─ 交易过少 → 优化选股范围
   │
   └─ 方案 B：切换策略库
       ├─ 趋势跟踪 → 均值回归
       ├─ 多因子 → 行业轮动
       └─ 返回策略自迭代重新验证
```

### 10.2 核心设计原则

> **风险调整后收益（Sharpe/Calmar）权重最高，回撤控制次之，绝对收益再次之。**

**理由**: A 股市场高波动、牛短熊长，2024 年初流动性危机教训表明**生存优先于收益**。

> **策略迭代必须形成闭环：回测→选股→条件单→监控→模拟→反馈→二次迭代**

**理由**: 策略参数不能拍脑袋决定，必须经过实盘数据模拟验证，验证结果必须反馈到回测系统进行二次优化。

> **风险调整后收益（Sharpe/Calmar）权重最高，回撤控制次之，绝对收益再次之。**

### 10.3 闭环设计说明（2026-03-31 新增）

**策略迭代不是线性流程，而是闭环**：

```
回测系统 → 策略自迭代 → 合格策略 → 选股 → 个股报告 → 条件单 → 监控池 → 模拟交易 → 验证 → 反馈 → 二次迭代
                                                                      ↑_______________________________________│
```

**关键设计点**：
1. **策略参数不能拍脑袋**：必须经过回测验证
2. **回测结果不能直接用**：必须经过实盘数据模拟验证
3. **模拟结果不能丢弃**：必须反馈到回测系统进行二次优化
4. **闭环形成**：反馈→二次迭代→重新验证，持续优化

### 10.4 各阶段详细说明（2026-03-31 修订）

#### 阶段 1：策略自迭代

**输入**: 用户设置初始参数（通过"🚀 一键自迭代"按钮）

**处理**:
- 保存初始配置为 v1 版本
- Optuna/网格搜索探索参数空间
- 运行回测
- 计算 7 项评分（Sharpe/最大回撤/年化/卡玛/盈亏比/胜率/换手率）
- 筛选高分配置保存为 v2/v3...

**输出**: 合格策略方案（评分≥0.75）

**关键 API**:
- `POST /api/iteration/start` - 启动迭代任务
- `GET /api/iteration/versions/:strategyType` - 获取版本历史

---

#### 阶段 2：应用到选股

**输入**: 合格策略方案（版本配置）

**处理**:
- 用户从策略库导入策略到选股页面
- 选股页面使用策略参数（四维度权重/七因子权重/PE/PEG 上限）
- 生成选股结果（Top3 行业 + Top10 股票池）
- 基于选股结果生成个股分析报告

**输出**: 选股结果 + 个股分析报告

**关键报告字段**:


**关键 API**:
- `GET /api/select` - 选股（使用策略参数）
- `POST /api/analysis/generate` - 生成个股分析报告

#### 阶段 6：验证策略实际表现

**输入**: 模拟绩效报告

**处理**:

### 10.4 综合评分计算公式

```python
def calculate_strategy_score(metrics):
    """计算策略综合评分"""
    config = STRATEGY_SCORING_CONFIG['metrics']
    scores = {}
    warnings = []  # 边界情况警告
    
    # 越大越好的指标
    higher_better = ['sharpe_ratio', 'annualized_return', 'calmar_ratio', 
                     'profit_loss_ratio', 'win_rate']
    
    for metric_name in higher_better:
        actual = metrics[metric_name]
        
        # 边界情况处理：盈亏比（全胜场景）
        if metric_name == 'profit_loss_ratio' and actual is None:
            # 全胜且样本不足 → 中性分 0.5，记录警告
            scores[metric_name] = 0.5
            warnings.append({
                'type': 'all_wins',
                'metric': 'profit_loss_ratio',
                'message': '策略全胜无亏损，盈亏比不可靠，建议扩大回测范围',
                'action_required': True
            })
        else:
            excellent = config[metric_name]['thresholds']['excellent']
            scores[metric_name] = min(actual / excellent, 1.0)
    
    # 越小越好的指标
    lower_better = ['max_drawdown', 'turnover_rate']
    
    for metric_name in lower_better:
        actual = abs(metrics[metric_name])
        danger = abs(config[metric_name]['thresholds']['danger'])
        scores[metric_name] = min(actual / danger, 1.0)
    
    # 加权计算
    total_score = sum(
        scores[metric_name] * config[metric_name]['weight']
        for metric_name in config
    )
    
    return total_score, scores, warnings
```

### 10.4.1 边界情况处理：盈亏比（全胜场景）

**场景**：策略回测结果全胜（无亏损交易）

**处理逻辑**：
```python
def calculate_profit_loss_ratio(trades):
    """计算盈亏比，处理全胜边界情况"""
    winning_trades = [t for t in trades if t.profit > 0]
    losing_trades = [t for t in trades if t.profit <= 0]
    
    if len(losing_trades) == 0:
        # 全胜场景
        return {
            'ratio': None,  # 不计入评分
            'is_all_wins': True,
            'win_count': len(winning_trades),
            'avg_win': sum(t.profit for t in winning_trades) / len(winning_trades),
            'warning': '策略全胜无亏损，盈亏比指标不可靠'
        }
    
    avg_win = sum(t.profit for t in winning_trades) / len(winning_trades)
    avg_loss = abs(sum(t.profit for t in losing_trades) / len(losing_trades))
    
    return {
        'ratio': avg_win / avg_loss if avg_loss > 0 else 10.0,
        'is_all_wins': False,
        'win_count': len(winning_trades),
        'lose_count': len(losing_trades),
        'avg_win': avg_win,
        'avg_loss': avg_loss
    }
```

#### 网格回测伪代码

```python
class GridBacktester:
    def __init__(self, upper, lower, grid_num, amount_per_grid):
        self.upper = upper  # 网格上限
        self.lower = lower  # 网格下限
        self.grid_size = (upper - lower) / grid_num
        self.levels = [lower + i * self.grid_size for i in range(grid_num + 1)]
        self.buy_orders = []   # 待成交买单队列
        self.sell_orders = []  # 待成交卖单队列
        self.pos = 0           # 总持仓
        self.balance = 100000  # 现金余额
        self.grid_profit = 0   # 已实现网格收益

    def on_bar(self, bar):
        """处理每一根 K 线"""
        
        # ========== 1. 撮合买单 ==========
        # 按价格从高到低排序（优先成交高价买单）
        for order in sorted(self.buy_orders, key=lambda x: x['price'], reverse=True):
            if bar.low <= order['price']:
                self.execute_buy(order)
                self.place_sell_order(order['price'] + self.grid_size)
                self.buy_orders.remove(order)

        # ========== 2. 撮合卖单 ==========
        # 按价格从低到高排序（优先成交低价卖单）
        for order in sorted(self.sell_orders, key=lambda x: x['price']):
            if bar.high >= order['price']:
                self.execute_sell(order)
                self.place_buy_order(order['price'] - self.grid_size)
                self.sell_orders.remove(order)
        
        # ========== 3. 破网处理 ==========
        if bar.close > self.upper:
            self.handle_breakout_upper(bar)
        elif bar.close < self.lower:
            self.handle_breakout_lower(bar)

    def execute_buy(self, order):
        """执行买入"""
        cost = order['price'] * order['qty']
        commission = max(5, cost * 0.00025)  # 万 2.5，最低 5 元
        if self.balance >= cost + commission:
            self.pos += order['qty']
            self.balance -= (cost + commission)
        else:
            # 资金不足，部分成交或失败
            pass

    def execute_sell(self, order):
        """执行卖出"""
        revenue = order['price'] * order['qty']
        stamp_duty = revenue * 0.001  # 印花税 0.1%
        commission = max(5, revenue * 0.00025)
        self.pos -= order['qty']
        self.balance += (revenue - stamp_duty - commission)
        self.grid_profit += (order['price'] - order['cost_price']) * order['qty']

    def calculate_total_profit(self, current_price):
        """计算总收益"""
        realized = self.grid_profit
        unrealized = self.pos * (current_price - self.avg_cost)
        return realized + unrealized
```

```python
# 日线回测网格的触发逻辑
def on_bar(self, bar):
    """
    bar 为日线数据
    bar.low = 当日最低价
    bar.high = 当日最高价
    """
    
    # 买单触发：当日最低价触及网格线
    for order in self.buy_orders:
        if bar.low <= order['price']:
            self.execute_buy(order)
            # 注意：日线数据下，一天可能只触发 1-2 个网格
            # 无法模拟分钟线下一天触发数十次的情况
    
    # 卖单触发：当日最高价触及网格线
    for order in self.sell_orders:
        if bar.high >= order['price']:
            self.execute_sell(order)
```

```python
def joint_optimization(base_equity, grid_alpha):
    """
    base_equity: 选股策略净值曲线（阶段 1 输出）
    grid_alpha: 网格超额收益曲线（阶段 2 输出）
    """
    best_w = 0
    best_sharpe = 0
    
    for w in np.linspace(0.5, 0.9, 20):  # 核心仓占比 50%-90%
        # 向量加权：总净值 = 核心仓 × w + 卫星仓 × (1-w)
        total_equity = base_equity * w + grid_alpha * (1 - w)
        
        sharpe = calculate_sharpe(total_equity)
        max_dd = calculate_max_drawdown(total_equity)
        
        if max_dd <= 0.20 and sharpe > best_sharpe:
            best_sharpe = sharpe
            best_w = w
    
    return best_w, best_sharpe
```

```python
}
```

| **商业化显现** ⭐ | 30% | 已商业化落地，非概念炒作 | 营收增速>20%、毛利率稳定、订单量 |
| **舆论热度** ⭐ | 25% | 媒体关注度、社交讨论度 | 新闻量、热搜指数 |
| **资本认可** | 20% | 机构持仓、融资流入 | 北向资金、公募持仓 |

**关键决策**：
> **商业化显现** 和 **舆论热度** 是核心中的核心
> - 商业化显现 = 有真实业绩支撑（避免"伪成长"）
> - 舆论热度 = 市场资金关注度

**个股筛选标准**：

| 维度 | 标准 |
|------|------|
| **基本面** | 营收增速>20%、毛利率稳定、经营现金流为正 |
| **增长潜力** | 研发投入占比>5%、市场份额提升、护城河清晰 |
| **估值** | PE/PEG 合理，非极端高估 |
| **技术面** | 中长期均线多头排列（MA20>MA60） |

**持股数量**：10 只（集中优势兵力）

#### 卫星仓（Satellite）20-30%

**核心理念**：与核心仓相同标的，用网格交易增厚收益

**卫星仓选股条件（必须同时满足）**：

| 条件 | 标准 | 理由 |
|------|------|------|
| **单笔交易≥3 万** | 避免最低 5 元佣金导致成本率>0.5% | 成本优化 ⭐ |
| **震荡区间稳定** | 近 3 个月振幅 20-50%，无极端单边行情 | 网格交易适用 |
| **总体发展可预见上行** | 满足核心仓选股标准 | 确保长期方向正确 |

**关键逻辑**：
> 卫星仓不是随意投机，而是**核心仓的延伸**
> - 用同一批优质标的
> - 在震荡期用网格交易降低成本
> - 长期方向向上，网格只赚波动的钱

**网格交易参数**：

| 参数 | 标准 | 理由 |
|------|------|------|
| **网格步长** | ≥0.8% | 覆盖印花税 0.1% + 佣金约 0.05% |
| **价格区间** | 近 3 个月高低点 | 避免破网风险 |
| **单笔金额** | ≥3 万 | 避免最低 5 元佣金 |

#### 核心 - 卫星协同机制

```
【选股阶段】
  └─ 4 维度行业筛选 + 7 因子个股评分
     └─ 输出：Top 10 股票池

【配置阶段】
  ├─ 核心仓（70-80%）：10 只股票，中长期持有
  └─ 卫星仓（20-30%）：从 10 只中选 3-5 只做网格

【执行阶段】
  ├─ 核心仓：双周频调仓，趋势跟踪
  └─ 卫星仓：网格交易，步长≥0.8%

【退出机制】
  └─ 核心仓卖出信号触发 → 同步清空对应卫星仓
     （避免"核心仓看空、卫星仓看多"的逻辑冲突）
```

---

#### 风控规则

**1. 网格交易成本红线**
```
单笔交易金额 ≥ 3 万
→ 佣金成本率 ≤ 0.025%（避免最低 5 元收费）
→ 网格步长 ≥ 0.8%（覆盖印花税 + 佣金）
```

**2. 趋势过滤规则**
```
卫星仓网格触发条件：
- MA20 > MA60：正常执行网格
- MA20 < MA60：暂停买入网格，只执行卖出
- 跌破网格下限 10%：停止网格，评估是否止损
```

**3. 核心仓与卫星仓联动**
```
核心仓卖出信号触发 → 同步清空对应卫星仓
```

### 15.2 回测选股数据来源（历史快照机制）

**共识编号**: DC-2026-03-24-002  
**优先级**: P0

#### 背景

回测系统必须避免"未来函数"问题：
- ❌ 错误做法：回测 2024-01-01 的股票选择，使用 2026-03-24 的最新数据
- ✅ 正确做法：回测 2024-01-01 的股票选择，使用 2024-01-01 或之前的历史数据

#### 核心原则

**回测时必须使用历史时点的快照数据**

#### 4 维度行业筛选 - 最终数据源方案

| 维度 | 数据源 | 历史可回溯 | 实施方案 |
|------|--------|-----------|---------|
| **政策支持** | `policy_events` 表 | ✅ | 手动录入 + 定时采集 |
| **商业化显现** | `fina_indicator` 表（财报） | ✅ | 查询时点前最新财报 |
| **舆论热度** | 市场关注度指标 | ✅ | 成交量 + 涨跌幅 + 换手率 |
| **资本认可** | 北向资金 + 机构持仓 | ✅ | 历史持仓数据 |

#### 舆论热度替代方案

**问题**：本地新闻数据库 83% 数据无 `pub_date`，无法用于历史回溯

**决策**：
- ❌ 放弃用本地新闻数据库做历史舆论热度
- ✅ 用**市场关注度**替代舆论热度

**市场关注度计算公式**：
```python
sentiment_score = 0.4 * volume_ratio + 0.4 * price_change + 0.2 * turnover_rate

其中：
- volume_ratio: 成交量变化（vs 前 5 日平均）
- price_change: 行业涨跌幅
- turnover_rate: 换手率
```

#### 历史快照表设计

## 十六、待实现事项（2026-03-31 新增）

### 16.1 模拟交易→反馈→二次迭代闭环

**状态**: ❌ 未实现

**缺失环节**：

| 环节 | 状态 | 说明 |
|------|------|------|
| 模拟交易执行 | ❌ 未实现 | 只有 execution_feedback 表记录实盘执行，没有主动模拟交易逻辑 |
| 验证策略实际表现 | ❌ 未实现 | 没有模拟绩效对比分析功能 |
| 反馈到回测系统 | ❌ 未实现 | 没有 backtest/feedback API |
| 二次迭代自动触发 | ❌ 未实现 | iteration-manager 有 simulationDeviation 检查，但没有自动触发二次迭代 |

**当前实现程度**：
```
回测 → 迭代 → 选股 → 报告 → 条件单 → 监控池 ✅
                                        ↓
                              模拟交易 → 验证 → 反馈 → 二次迭代 ❌
```

**设计思路**（待讨论）：

#### 方案 A：轻量级模拟（推荐）

**核心思路**：利用现有监控池 + 条件单执行记录，不单独建模拟账户

**实现方式**：
1. 策略版本发布时，标记为"模拟验证中"
2. 监控池条件单触发时，同时写入 `execution_feedback` 表
3. 定期（每日/每周）聚合 execution_feedback 数据，计算模拟绩效
4. 对比模拟绩效 vs 回测绩效，生成偏差报告
5. 偏差>20% 时，自动触发二次迭代任务

**优点**：
- 复用现有表结构（execution_feedback）
- 不需要额外的模拟交易引擎
- 数据真实（基于实盘数据触发）

**缺点**：
- 无法模拟滑点/成本（execution_feedback 记录的是真实成交价）
- 需要等待真实触发，验证周期长

---

#### 方案 B：独立模拟账户

**核心思路**：创建独立的模拟账户系统，与实盘完全隔离

**实现方式**：
1. 创建 `mock_account`、`mock_position`、`mock_trade` 表
2. 监控池条件单触发时，同时执行实盘条件单和模拟交易
3. 模拟交易使用实时行情 + 滑点假设计算成交价
4. 定期计算模拟绩效，对比回测结果
5. 偏差>20% 时，触发二次迭代

**数据库设计**：
```sql
-- 模拟账户表
CREATE TABLE mock_account (
    account_id TEXT PRIMARY KEY,
    strategy_version_id TEXT,
    initial_capital REAL,
    current_capital REAL,
    created_at TEXT
);

-- 模拟持仓表
CREATE TABLE mock_position (
    position_id TEXT PRIMARY KEY,
    account_id TEXT,
    ts_code TEXT,
    quantity INTEGER,
    avg_cost REAL,
    created_at TEXT
);

-- 模拟交易表
CREATE TABLE mock_trade (
    trade_id TEXT PRIMARY KEY,
    account_id TEXT,
    ts_code TEXT,
    action TEXT,
    quantity INTEGER,
    price REAL,
    simulated_price REAL,  -- 考虑滑点后的价格
    slippage REAL,
    commission REAL,
    stamp_duty REAL,
    pnl REAL,
    occurred_at TEXT
);

-- 模拟绩效表
CREATE TABLE mock_performance (
    performance_id TEXT PRIMARY KEY,
    account_id TEXT,
    period_start TEXT,
    period_end TEXT,
    total_return REAL,
    annualized_return REAL,
    max_drawdown REAL,
    sharpe_ratio REAL,
    trade_count INTEGER,
    win_rate REAL,
    avg_slippage REAL,
    backtest_deviation REAL,  -- 与回测的偏差
    created_at TEXT
);
```

**优点**：
- 完全隔离，不影响实盘
- 可以自定义滑点/成本假设
- 可以快速验证（可以用历史数据回测模拟）

**缺点**：
- 需要新建表结构和 API
- 开发工作量大

---

#### 方案 C：历史数据回放模拟

**核心思路**：用历史分钟线数据回放策略执行，不依赖实时数据

**实现方式**：
1. 选择一段历史区间（如最近 3 个月）
2. 用分钟线数据回放条件单触发逻辑
3. 计算模拟成交价（考虑滑点）
4. 生成模拟绩效报告
5. 对比回测结果

**优点**：
- 不需要等待真实触发
- 可以快速验证多个策略版本
- 可以重复回测同一段数据

**缺点**：
- 不是真正的"实盘数据验证"
- 无法验证策略对突发事件的响应

---

### 推荐方案：方案 A + 方案 C 组合

**短期**（方案 C）：先用历史数据回放模拟，快速验证策略
**长期**（方案 A）：积累实盘执行数据，持续优化策略

**理由**：
1. 方案 C 可以快速填补闭环空白
2. 方案 A 可以长期积累真实数据
3. 两者互补，不冲突

---

### 待决策事项

| 事项 | 选项 | 建议 |
|------|------|------|
| 模拟交易实现方式 | 方案 A/B/C | 方案 A+C 组合 |
| 反馈触发条件 | 定期/事件驱动 | 定期（每日）+ 偏差>20% 事件驱动 |
| 二次迭代触发 | 自动/手动 | 偏差>20% 自动触发，人工确认 |
| 优先级 | P0/P1/P2 | P1（核心闭环已通，此为优化） |

## 十七、四维度七因子策略优化设计（2026-03-29 新增）

### 17.1 目标

把"四维度七因子策略"的自动优化，从当前的"只调少量交易阈值"，升级成一套可渐进落地的分层优化方案。

**本方案优先解决**：
1. 当前优化对象不完整
2. 权重参数未进入搜索空间
3. 高分结果缺乏样本外验证
4. 回测分、执行反馈、策略发布条件口径混杂

**本方案不做**：
1. 不引入 HMM / RL / Transformer 这类重研究型改造
2. 不推翻现有 Node.js + SQLite + Python Optuna 架构
3. 不在第一阶段直接把所有参数一次性放进大搜索空间

### 17.2 分层优化架构

**三层主优化 + 一层验证增强**：

| 层级 | 职责 | 参数归属 |
|------|------|----------|
| **筛选层** | 决定"哪些股票进入候选池" | min_score, pe_max, peg_max, max_price |
| **权重层** | 决定"候选股票如何被评分" | 四维度权重、七因子权重 |
| **交易层** | 决定"筛出股票后怎么进出场" | score_stop_loss, stop_loss, take_profit |
| **验证层** | 样本外验证增强 | IS/OOS 窗口、WFE 指标 |

### 17.3 参数分层归属

**筛选层参数**：
- `min_score`: 0.65 ~ 0.90
- `pe_max`: 30 ~ 100
- `peg_max`: 1.0 ~ 4.0
- `max_price`: 80 ~ 800

**权重层参数**（四维度）：
- `dimensionWeights.social`: 社会经济趋势权重
- `dimensionWeights.policy`: 政策方向权重
- `dimensionWeights.public`: 舆论热度权重
- `dimensionWeights.business`: 商业变现权重
- 归一化约束：sum = 1.0

**权重层参数**（七因子）- P1 阶段：
- `factorWeights.trend/momentum/valuation/earnings/capital/volatility/sentiment`
- 防极端约束：单因子权重 0.03 ~ 0.40

**交易层参数**：
- `score_stop_loss`: 0.50 ~ 0.80
- `stop_loss`: 0.03 ~ 0.15
- `take_profit`: 0.10 ~ 0.50

**网格参数** - P1 独立优化：
- `gridConfig.gridSize`
- `gridConfig.maxPosition`
- `gridConfig.triggerThreshold`

### 17.4 实施计划

**P0：立即落地**
1. 扩展 seven_factor 搜索空间（7 个参数）
2. 接入四维度权重搜索（4 维权重）
3. 版本历史补充样本内指标口径
4. 加入最小样本外验证（训练 2023-2024.06，验证 2024.07-12）

**P1：中期改造**
1. 接入七因子权重搜索（7 维权重）
2. 多目标评分函数（交易笔数、换手惩罚）
3. 网格参数独立自动优化

**P2：后续增强**
1. 真正的 Walk-Forward 分析
2. 市场状态切换（规则法）
3. 因子稳定性指标（IC、IC 衰减）

### 17.5 权重优化技术选型

**第一版推荐**：归一化系数
- 采样 4 个非负系数 a,b,c,d
- 归一化：w_i = a_i / sum(a)
- 优点：易读易调试，与 Optuna 兼容

**第二版可选**：Dirichlet 采样
- 适合权重搜索成熟后
- 更标准的 simplex 采样

**当前不推荐**：Softmax 温度搜索
- 解释性不够直接
- 增加额外调试成本

## 十八、独立模拟账户设计（2026-04-01 新增）

### 18.1 核心目标

创建独立的模拟账户系统，与实盘完全隔离，用于：
1. 验证策略实际表现（回测 vs 模拟）
2. 计算绩效偏差
3. 触发二次迭代（偏差>20%）

### 18.2 系统架构

```
策略版本发布 → 创建模拟账户 → 关联 strategy_version_id
                    ↓
监控池条件单触发 → 同时执行：a) 实盘条件单  b) 模拟交易
                    ↓
模拟交易引擎 → 获取实时行情 → 应用滑点 (0.1%) → 写入 mock_trade
                    ↓
绩效计算（每日）→ 聚合 mock_trade → 计算指标 → 对比回测
                    ↓
偏差检测 → 偏差>20% → 自动触发二次迭代
```

### 18.3 数据库设计（4 张新表）

**mock_account** - 模拟账户表
- account_id, strategy_version_id, account_name
- initial_capital, current_capital, available_capital
- status (active/stopped/closed), created_at, started_at, stopped_at

**mock_position** - 模拟持仓表
- position_id, account_id, ts_code
- quantity, avg_cost, current_price, market_value, unrealized_pnl

**mock_trade** - 模拟交易表
- trade_id, account_id, ts_code, action (BUY/SELL)
- price (行情价), simulated_price (考虑滑点), slippage_rate
- commission, stamp_duty, pnl, trigger_source, strategy_version_id

**mock_performance** - 模拟绩效表
- performance_id, account_id, strategy_version_id
- total_return, annualized_return, max_drawdown, sharpe_ratio
- trade_count, win_rate, profit_loss_ratio
- backtest_total_return, backtest_deviation, is_deviation_exceeded

### 18.4 核心算法

**模拟成交价计算**：
```javascript
// 买入：价格上浮，卖出：价格下浮
if (action === 'BUY') return marketPrice * (1 + slippageRate);
else return marketPrice * (1 - slippageRate);
```

**交易成本**：
- 佣金：万分之 2.5，最低 5 元
- 印花税：卖出时千分之 1

**偏差阈值**：20%

### 18.5 API 设计

| API | 方法 | 说明 |
|-----|------|------|
| /api/mock/account/create | POST | 创建模拟账户 |
| /api/mock/account/list | GET | 账户列表 |
| /api/mock/account/stop | POST | 停止账户 |
| /api/mock/trade/execute | POST | 执行模拟交易 |
| /api/mock/trade/list | GET | 交易记录 |
| /api/mock/performance/current | GET | 当前绩效 |
| /api/mock/performance/deviation | GET | 偏差超标账户 |
| /api/iteration/trigger-by-deviation | POST | 触发二次迭代 |

### 18.6 实施计划

**Phase 1 (P0, 9h)**: 数据库迁移 + 模拟交易引擎 + 绩效计算
**Phase 2 (P1, 9h)**: 偏差检测 + 账户管理 API + 前端页面
