# 2026-03-27 系统总闭环共识

**创建时间**: 2026-03-27 10:15  
**参与**: 主人、Codex  
**主题**: 执行流程与研究流程的系统级闭环重构

---

## 1. 背景

当前项目已经同时具备：
- 选股分析、个股分析、监控池、条件单
- 回测系统、策略迭代、策略模板/策略库

但系统心智仍然不够清晰，主要问题是：

1. **选股分析页与回测系统边界不够清楚**
   - 用户容易误以为“选股页内直接一键回测”是主路径
2. **研究流程与执行流程关系不清**
   - 页面存在功能，但没有被统一解释成一个大闭环
3. **策略库尚未成为唯一中枢**
   - 还存在“回测策略 / 选股策略 / 模板配置”多套表述

---

## 2. 最终共识

系统不是两条完全独立的线，而是一个**总体闭环**：

```text
回测 -> 策略管理/迭代 -> 策略库 -> 选股分析 -> 个股分析 -> 监控池 -> 条件单 -> 执行反馈 -> 回测/策略管理
```

其中包含两条子流程：

### 2.1 研究流程

```text
回测系统 -> 策略管理/迭代 -> 高评分策略 -> 策略库
```

职责：
- 研究策略
- 回测验证
- 策略评分
- 参数优化
- 版本管理
- 形成可发布策略

### 2.2 执行流程

```text
选股分析 -> 个股分析 -> 监控池 -> 条件单 -> 执行跟踪
```

职责：
- 使用可发布策略进行选股
- 生成人工/自动可执行的个股决策
- 进入监控与条件单执行链路
- 产生日后可复盘的执行反馈

---

## 3. 两条流程的关系

### 3.1 前向传递

研究流程的输出，不直接等于“开始交易”，而是先进入策略库：

```text
回测评分通过 -> 进入策略库 -> 选股分析页导入参数 -> 执行流程
```

### 3.2 反馈回流

执行流程不是终点，执行结果必须反向进入研究流程：

```text
条件单执行结果 / 监控表现 / 实盘或模拟绩效
    -> 回流到回测系统与策略管理
    -> 作为下一轮参数迭代和策略切换依据
```

### 3.3 核心中枢

**策略库是两条流程的唯一中枢。**

策略库同时承担三种角色：
- **研究流程产出中心**：保存高评分策略、版本号、参数、适用场景
- **执行流程消费入口**：选股分析页从这里导入策略参数
- **反馈归档中心**：记录哪些策略真正进入执行链、执行表现如何

## 4. 页面职责重定义

### 4.1 选股分析页 `select.html`

定位：**执行流程入口**

只负责：
- 从策略库导入策略参数
- 配置选股参数
- 生成选股结果
- 进入个股分析

不负责：
- 承担回测心智
- 作为策略研究入口
- 引导用户直接做策略迭代

### 4.2 个股分析页 `analysis.html`

定位：**执行决策页**

只负责：
- 对选出的个股做更深分析
- 输出结构化策略建议
- 导向监控池或条件单

### 4.3 监控池 `monitor-pool.html`

定位：**执行跟踪页**

只负责：
- 跟踪进入执行观察名单的股票
- 结合策略建议和实时状态生成监控信号

### 4.4 条件单 `conditional-order.html`

定位：**执行动作页**

只负责：
- 以条件单形式表达策略执行
- 记录触发、取消、成交、失效等执行反馈

### 4.5 回测系统 `backtest.html`

定位：**研究流程入口**

只负责：
- 历史验证
- 参数试验
- 策略评分
- 生成可进入策略库的候选策略

### 4.6 策略迭代页 `iteration-manager.html`

定位：**研究流程深化页**

只负责：
- 自动迭代
- 版本管理
- 策略对比
- 策略发布到策略库

## 5. 文案与交互原则

### 5.1 选股页文案

选股页应强调：
- “导入策略库策略”
- “导入策略参数”
- “保存当前选股方案”

选股页不应强调：
- “导入回测策略”
- “一键回测”
- “直接去迭代”

### 5.2 回测页文案

回测页应强调：
- “策略研究”
- “回测验证”
- “一键自迭代”
- “发布到策略库”

### 5.3 迭代页文案

迭代页应强调：
- “策略版本”
- “评分变化”
- “优化结果”
- “发布策略”

---

## 6. 数据闭环要求

### 6.1 研究流程输出到策略库

至少要保存：
- strategy_id
- version
- strategy_type
- params
- score
- backtest_metrics
- created_at
- source = backtest / iteration

### 6.2 执行流程消费策略库

选股分析页至少要能读取：
- strategy_id
- strategy_type
- params
- compatible_with = selection

### 6.3 执行反馈回流研究流程

至少要能回流：
- 使用的是哪个 strategy_id / version
- 进入了哪些股票
- 条件单触发率
- 命中率 / 盈亏 / 失效原因
- 监控池中预警表现

这些反馈不是立即替代回测，而是作为研究流程的真实世界校正信号。

---

## 7. 当前阶段的实现优先级

### P0：流程边界澄清
- 收敛页面文案
- 收敛按钮文案
- 收敛“模板 / 策略库 / 参数导入”表述

### P1：策略库中枢化
- 统一策略库的入口与命名
- 明确研究流程发布、执行流程导入

### P2：执行反馈回流
- 设计条件单/监控结果的反馈记录结构
- 明确如何在策略管理中展示这些反馈

---

## 8. 一句话结论

系统最终不是“选股页”和“回测页”各做各的，而是：

**研究流程产出策略，策略库沉淀策略，执行流程验证策略，执行反馈再反哺研究流程。**
